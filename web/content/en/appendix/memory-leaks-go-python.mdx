# Memory leak patterns (Go & Python)

A **memory leak** (in the everyday engineering sense) is **memory that is no longer logically needed** but remains **reachable**, so reclaimers cannot free it. Usage climbs monotonically until latency spikes or the process hits **OOM**.

This appendix mirrors the Chinese version: **Go mechanics**, a **short cross-language table**, **Python mechanics**, and **detection** ideas.

---

## 1. Typical Go patterns

### 1.1 Goroutine leaks

```go
func handleRequest() {
	ch := make(chan int)

	go func() {
		val := <-ch // blocks forever if nothing is sent
		println(val)
	}()

	// Function returns; nothing will ever send on ch → goroutine never exits
}
```

**Fix**: pair send/receive, `close(ch)`, buffered channels, timeouts via `select`, or cancellation with `context`.

### 1.2 Unbounded global maps

```go
var cache = make(map[string][]byte)

func process(data []byte) {
	key := generateKey(data)
	cache[key] = data // never evicts
}
```

**Fix**: LRU/TTL, periodic pruning, or an external store with a cap.

### 1.3 Closures and the loop variable (Go 1.22+ semantics)

```go
func createHandlers() []func() {
	var handlers []func()
	for i := 0; i < 10; i++ {
		handlers = append(handlers, func() {
			println(i) // Go ≤1.21: often shared `i`; Go ≥1.22: per-iteration variable
		})
	}
	return handlers
}
```

Since **Go 1.22**, each iteration gets a **fresh** loop variable; older releases need `i := i` inside the loop. Large values captured by long-lived closures can still pin memory.

### 1.4 `http.Response.Body` not closed

```go
func fetchData(url string) {
	resp, _ := http.Get(url)
	// Missing: defer resp.Body.Close()
	body, _ := io.ReadAll(resp.Body)
	println(len(body))
}
```

**Fix**: always `defer resp.Body.Close()` (and handle errors from `Get`/`ReadAll` in real code).

### 1.5 `time.Ticker` not stopped

If nothing calls `ticker.Stop()` and the receive loop cannot exit, the ticker goroutine path never winds down.

## 2. Quick comparisons (other languages)

| Language | Pattern | Root cause |
| :--- | :--- | :--- |
| **C/C++** | `malloc` without `free` | Manual pairing omitted |
| **Java** | static collections, thread-locals | strong references never cleared |
| **JavaScript** | listeners, DOM closures | host graph stays reachable |
| **Python** | globals, cycles + `__del__`, C extensions | see below |

## 4. Investigating Go

| Tool | Role |
| :--- | :--- |
| **`net/http/pprof`** | `heap`, `goroutine`; goroutine stacks expose stuck workers |
| **`go tool pprof`** | heap profiles, inuse vs alloc |
| **`runtime.ReadMemStats`** | process-level counters |

**Red flags**: flat traffic but rising RSS/heap; goroutine count monotone up; more GC work without heap returning.

**One-liner (Go)**  
> Most “Go memory leaks” are really **leaked concurrency or connections**; the heap follows.

## 5. Python: runtime and patterns

### 5.1 Reference counting + cyclic GC

CPython frees most objects when **refcount hits zero**; **reference cycles** wait for the **generational** collector. Pure cycles **without** `__del__` are usually collected after external references drop—but **later** and with extra cost.

### 5.2 Breaking cycles (example)

Use **`weakref`** on one edge (e.g. parent → child strong, child → parent weak) or explicit cleanup routines.

### 5.3 `__del__` and cycles

Defining `__del__` on objects that participate in cycles is fragile: ordering and timeliness are problematic. Prefer **`with` / context managers** or **`weakref.finalize`**.

### 5.4 Globals and class-level lists

Class attributes that accumulate callbacks without removal behave like unbounded caches.

### 5.5 C extensions

Native code may allocate off-heap buffers; Python objects may disappear while **RSS** still grows—use allocator tooling and vendor docs.

### 5.6 Generators

Call **`gen.close()`** (or exhaust the iterator) so frames release large temporaries promptly.

| :--- | :--- |

| **`gc`** | `collect()`, introspection (`garbage` is for debugging) |

## 7. Python vs Go

| Aspect | Python | Go |
| :--- | :--- | :--- |
| **Usual suspects** | global caches, long-lived containers, `__del__`+cycles, C libs | goroutines, channels, tickers, HTTP bodies, global maps |
| **Telemetry** | RSS, `tracemalloc`, ref graphs | pprof heap & goroutine |
| **Mental model** | refcount + generational GC | concurrency + explicit resource teardown |

**One-liner (Python)**  
> Most “leaks” are **unexpected strong references**; cycles plus `__del__` are the painful corner.
