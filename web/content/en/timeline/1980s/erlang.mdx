## Origin Story

Erlang was created at Ericsson in 1986 by Joe Armstrong to solve a critical problem: telephone switches required near-perfect uptime (99.999% availability, or less than 5 minutes downtime per year). Traditional languages couldn't handle the demands of massive concurrency while allowing hot code updates without system restart.

Armstrong designed Erlang specifically for telecommunications infrastructure, where systems must run continuously for decades. The language's name originally stood for "Ericsson Language" but was later redefined as a tribute to the mathematician Agner Krarup Erlang, who pioneered traffic engineering.

The BEAM virtual machine (Bogdan/Björn's Erlang Abstract Machine) that runs Erlang was designed for extreme concurrency from day one, supporting millions of lightweight processes on a single machine.

## Design Philosophy

Erlang's core philosophy centers on "let it crash"—rather than trying to handle every possible error, the system is designed to quickly detect failures and restart failed components. This approach, called OTP (Open Telecom Platform), revolutionized how fault-tolerant systems are built.

The Actor model is fundamental: each Erlang process is a lightweight isolated actor that communicates exclusively through message passing. Processes share no memory, eliminating entire categories of bugs common in threaded systems.

Hot code loading allows Erlang systems to update their own code while running, without stopping or interrupting service. This was revolutionary for systems that cannot be taken offline.

## Code Example

```erlang filename="Erlang Example"
%% Concurrent factorial server
-module(factorial).
-export([start/0, calc/2]).

start() ->
    spawn(fun() -> server_loop(1) end).

calc(Pid, N) ->
    Pid ! {self(), N},
    receive
        {Pid, Result} -> Result
    end.

server_loop(N) ->
    receive
        {From, 0} -> From ! {self(), N}, server_loop(N);
        {From, N} when N > 0 -> server_loop(N * (N - 1))
    end.

%% Spawning 1 million lightweight processes
spawn_processes(0) -> done;
spawn_processes(N) ->
    spawn(fun() -> receive after infinity -> ok end end),
    spawn_processes(N - 1).
```

## When to Use

**Good fit:**
- Telecommunications infrastructure
- Messaging systems requiring 99.999% uptime
- Systems needing hot code updates
- Massive concurrent connections (millions of WebSocket connections)

**Not a good fit:**
- CPU-intensive computation (better: C, Rust)
- Simple scripts (better: Python, Ruby)
- Object-oriented business logic (better: Java, C#)

## Controversies

<Callout type="warning">
  Erlang's syntax, heavily influenced by Prolog, uses single assignment variables and pattern matching extensively. Developers accustomed to imperative languages often find the syntax alien. Variables are assigned once and never changed—a foreign concept that becomes natural but initially confusing.
</Callout>
