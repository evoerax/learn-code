# 总线与通信

现代嵌入式机器是多个子系统的集合，这些子系统必须实时共享状态、指令和传感器数据。选择正确的总线或链路协议，会直接影响延迟、确定性、接线复杂度乃至法规合规性。本页从芯片间板级通信讲起，覆盖遥控、数传、ROS/MQTT/DDS 中间件，最后给出选型与实践建议。

## 板级与机内协议

这类协议位于同一机壳内，通常在同一 PCB 或短背板上。

**UART / SPI / I2C** 是 MCU 连接传感器和执行器芯片的三种主力方案。UART（通用异步收发器）为点对点，只需两根信号线（TX/RX）加地线，是 GPS 模块、ESP32 遥测和调试串口的经典选择。SPI（串行外设接口）为同步全双工，支持更高吞吐率，适合高速 ADC 采样、外部 Flash 或显示屏。I2C（集成电路间总线）仅用两根共享线（SDA/SCL），支持多主机多从机，是读取磁力计、气压计及部分 IMU 的标准方式。三者构成嵌入式入门的必修三件套，覆盖机器人和车辆内部绝大多数芯片间链路。

**CAN（控制器局域网）** 是汽车和工业机器人领域的主导总线。它是双线差分总线，内置仲裁机制（ID 优先级高的消息优先发送，不破坏数据），具备错误检测与重传能力，在短总线长度下支持约 5 Mbps。CAN 消息以数值 ID 标识，而非 I2C 的从机地址，这使得添加节点无需中央协调器。总线空闲时每个节点均可发送，碰撞由 ID 优先级解决——这使 CAN 具有确定性，适用于电机关闭、制动等安全关键功能。CAN 同时需要 **120 Ω 终端电阻** 接在总线两端，以消除信号反射；终端电阻缺失会导致边沿振铃，引发偶发性错误帧。

> **CAN 时钟与终端电阻实践**
> - **时钟精度**：CAN 控制器使用外部石英晶振或内部 RC 振荡器。石英精度通常为 ±50 ppm，足以满足 1 Mbps 以下的位时序要求；RC 振荡器精度仅 ±1%~±2%，会导致高速率（>500 kbps）下位时序偏移，引发采样点错位和错误帧。建议对所有节点使用相同标称频率的石英晶体，并在总线两侧各串联一个 120 Ω 终端电阻（有些长总线会采用 120 Ω + 60 Ω 分布式端接）。测量终端电阻时用万用表在 CANH-CANL 间直接读取，应接近 60 Ω（两节点各 120 Ω 并联）。
> - **终端电阻位置**：必须在总线物理两端，不能放在中间节点。汽车上常见的错误是把终端电阻放在 ECU 壳体内而非真正的总线终点（总线走向最远的两个节点处），这会导致中间段阻抗失配，在长线束上产生反射。
> - **排查技巧**：若 CAN 网络出现偶发错误帧（error passive），先用示波器在总线远端观察波形——上升沿/下降沿应干净无过冲；若观察到振铃，在终端电阻上串联一个小的阻尼电阻（10~47 Ω）通常可以改善。

**以太网 / USB** 位于高带宽端。以太网（100BASE-T 或带外部 PHY 的微控制器千兆以太网）可提供数十至数百 Mbps 吞吐率，是大型系统中摄像头、激光雷达、计算模组（SOM/NUC）和调试接口的自然选择。USB 2.0/3.0 在设备已具备 USB 接口时承担类似角色——网络摄像头、固态存储和高速 ADC 是常见例子。两者都需要注意连接器选型，以太网还需磁耦和精确阻抗控制的布线来维持信号完整性。

## 链路层：遥控与数传

信号必须通过空气而非铜线传播时，延迟、带宽、法规合规和安全性成为主要约束。

**遥控接收（RC）** 仍是无人机和 hobby 车辆的主要安全链路。PWM 接收机为每个通道输出离散脉冲（通常 1~2 ms 宽，50 Hz 重复），飞控用输入捕获定时器读取这些脉冲以确定飞手意图。现代 RC 协议如 SBUS、CRSF（Crossfire）和 FPort 将多通道编码为串行数据流，仅需一根信号线即可支持更高更新率（SBUS 最高 1 kHz）。无论何种协议，必须配置失控保护：信号丢失时，车辆应进入预设安全状态（定高保持、返航或电机切断）。

**数传与图传** 完善了无线通信的全貌。数据数传（915 MHz 或 433 MHz 无线电台，使用 MAVLink、LTM 或 FrSky SmartPort 协议）将实时遥测数据——电池电压、GPS 位置、实时电流——传输至地面站。图传（5.8 GHz 模拟图传或 DJI O3、HDZero、Walksnail 等数字系统）向飞手传输高带宽视频。两者均受当地无线电法规约束：发射功率限制、可用频段和许可要求各国不同。在部署新的无线链路前，尤其是商业销售需要 CE/FCC 认证的产品，务必核实合规性。

## 中间件：ROS 主题、MQTT 与 DDS

在系统层面，进程间或板间通信从原始点对点链路升级为结构化的发布/订阅或服务层。

**ROS 主题** 提供发布/订阅消息层。节点向命名主题发布消息，其他节点订阅接收。主题有类型定义（如 `sensor_msgs/Imu`、`geometry_msgs/Twist`），这在生产者和消费者之间建立了契约，并支持 `rostopic echo` 等调试工具。ROS 1 依赖单一主节点进行名称注册，这引入了单点故障。ROS 2 用 DDS（数据分发服务）替代了主节点——DDS 是标准化的点对点中间件，提供自动发现、可配置 QoS（服务质量）策略，并原生支持实时系统。

**MQTT（消息队列遥测传输）** 是为不稳定网络设计的轻量级发布/订阅协议，常见于 IoT 网关和云端连联车辆。它使用 broker 中转消息，客户端发布到主题（如 `/vehicle/left_motor/current`）并订阅所需主题。MQTT 的三级 QoS（0：最多一次，1：至少一次，2：恰好一次）让开发者可以在带宽、延迟和交付保证之间权衡。对于向云端仪表盘推送遥测同时接收指令的机器人，MQTT 通常是自然选择。

**DDS（数据分发服务）** 是 ROS 2 的底层协议。DDS 是 OMG 标准化规范，有多家实现（Fast DDS、Cyclone DDS、RTI Connext）。DDS 支持细粒度 QoS 策略，包括截止时间监控、存活检测、所有权强度（用于冗余发布者）和传输优先级。对于多车辆或多机器人部署——延迟、可靠性和实时保证至关重要——DDS 是专业级选择。

## 设计要点

无论选择何种总线或链路，以下三个系统级属性值得特别关注：

**带宽与更新率** 必须与控制回路匹配。以 1 kHz 运行的飞控需要至少同频率或更快的传感器数据刷新；以 10 Hz 发布 IMU 数据会造成根本性的控制问题。将每条总线的实际吞吐率与系统所需数据率对应，留出协议开销和重传的余量。

**延迟与抖动** 对闭环控制影响最大。UART 延迟通常低于 1 ms，但在 OS 调度下非确定性。CAN 最坏情况延迟取决于总线负载和消息优先级——低优先级长帧后会延迟数百微秒。对于「传感器→估计→控制→执行」硬实时链路，逐一检查每条链路的抖动预算是否在控制回路周期内。

**文档化与接口契约** 避免样机「只有一个人能飞/能开」的问题。在原理图中标注每条总线、连接器和协议版本；注明 CAN 比特率和终端电阻状态；记录每个节点发布和订阅的主题或消息类型，维护单一真相来源（共享 `.msg` 或 `.srv` 定义、ROS bag 索引或 MQTT 主题树）。任何工程师都能探针总线并读取数据的系统，远比需要隐性知识的系统更可维护。

## 整机关系

三类整机强调通信栈的不同部分。无人机重度依赖 RC（SBUS/CRSF 飞手输入）和轻量数传（MAVLink over 433 MHz），更大有效载荷的多旋翼平台上 CAN 正越来越常见用于跨控制器通信。汽车以 CAN 为主（OBD-II、诊断、ECU），ADAS 传感器和娱乐系统用汽车以太网（100BASE-T1、1000BASE-T1），低带宽传感器如车窗开关用 LIN。工业机械臂和 AMR 使用 EtherCAT 或 PROFINET 实现硬实时关节控制，ROS 2/DDS 用于感知-计算通信，MQTT 用于向上位机仪表盘的车队级状态上报。

## 实践建议

从 UART 开始：任何带 UART 接口的传感器（GPS、部分激光雷达、Betaflight 风格 ESC 遥测）默认都是 UART，只需 TX/RX 交叉连接即可工作。进阶到 CAN 当你需要多节点确定性、电气隔离或差分信号的鲁棒性时。对于 ROS 系统，从 ROS 2 Humble 和 Fast DDS 入手——学习曲线比看起来平缓，工具链（rviz2、ros2 topic、ros2 doctor）值得投入。云端连联车队先用 MQTT over Wi-Fi 原型验证，再决定是否上蜂窝调制解调器或 LoRa——在不可靠无线上调试比在稳定 LAN 上调试难度高得多。
