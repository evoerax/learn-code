# 2020s — 默认安全

2020 年代仍在书写之中。但方向已经清晰：这个十年涌现的语言将内存安全、类型安全和正确性视为默认值——而非选项。一些语言在系统编程领域挑战 C 和 C++（Zig、Carbon、Mojo），另一些将类型安全带入区块链（Move）和分布式系统（Roc）等新领域。"没有替代品所以只能用不安全语言"的时代正在终结。

## 背景

| | |
|---|---|
| **时代特征** | 后疫情远程工作、AI 热潮、WebAssembly 成熟 |
| **重要事件** | Rust 进入 Linux 内核（2022）、NSA/CISA 内存安全建议（2022–2023）、GPT-4（2023） |
| **里程碑** | 美国政府机构建议对新代码使用内存安全语言 |
| **核心挑战** | 在安全关键系统中替换 C/C++；AI 辅助编程；区块链正确性 |

<Callout type="info">
  2022 年，NSA 发布指导意见，建议各组织向内存安全编程语言过渡，并将 Rust、Go、Swift、Java 等列为优于 C 和 C++ 的首选语言。
</Callout>

## 核心主题

**C 的继任者们相互竞争。** 在"系统编程离不开 C"几十年之后，现在存在多个可行的替代方案。Rust 已经进入 Linux 内核。Zig 提供了没有未定义行为的 C 级控制。Carbon 被设计为可以被现有 C++ 代码库逐步采用。Mojo 为 AI 工作负载扩展 Python，使其具备系统级能力。单一文化正在瓦解。

**comptime 取代元编程。** Zig 的 `comptime`——编译时计算——是 C 预处理器宏和 C++ 模板元编程的简洁替代品。在编译时运行的代码与运行时代码使用相同的语言；没有单独的宏语言需要学习。这种模式影响了对其他语言编译时能力的思考。

**AI 改变了语言的使用方式。** 大型语言模型可以用任何语言编写代码，但对语法清晰、一致且训练语料充足的语言效果最好。这产生了新的压力：语言设计必须考虑 LLM 的可读性，而不仅仅是人类可读性。Mojo 明确针对 AI/ML 工作负载，将 Python 的生态系统与系统性能结合起来。

**区块链要求正确性。** Move 是为 Diem（前 Libra）区块链设计的，核心洞见是：数字资产应该被视为不能被意外复制或销毁的资源。Move 的类型系统在编译时强制执行这一点。在 Solidity 中损失数百万的智能合约漏洞，在 Move 中成为编译时错误。

**简洁作为特性。** Hare 和 Zig 拒绝"魔法"——隐式行为、隐藏分配、复杂的元编程。其哲学是：程序员应该能够阅读代码并准确理解机器将做什么，而无需了解编译器变换。这是对 C++ 和 Java 中积累的复杂性的反应。

## 悬而未决的问题

2020 年代的编程语言格局面临着前几十年未曾提出的问题：

- **AI 结对编程**：当你的协作者是语言模型时，语言设计是否需要改变？
- **WebAssembly 作为通用目标**：WASM 能否成为新的 JVM——一个任何语言都可以使用的可移植沙盒执行环境？
- **大规模形式验证**：证明辅助工具能否从学术工具转变为日常编程实践？
- **能源效率**：随着数据中心消耗越来越多的电力，性能和效率方面的考量是否会改变语言的选择？

这些问题的答案将塑造 2030 年代的面貌。
