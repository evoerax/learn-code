# 编程范式

编程范式是一种基本的编程风格或方法——一套关于程序应如何组织、计算应如何表达、以及抽象单元应是什么的思想体系。大多数语言支持多种范式；下面的分类反映了每种语言的主要设计理念。

<Callout type="info">
  没有哪种范式是普遍优越的。每种范式都刻画了一种不同的计算思维模型，并在各自擅长的领域表现突出。最具影响力的语言往往是开创或综合范式的先行者，而不是僵化地实现某一种范式。
</Callout>

## 主要范式一览

| 范式 | 核心思想 | 代表语言 |
|------|---------|---------|
| 命令式 | 描述*如何做*：一步步改变程序状态的指令序列 | Assembly、C、Pascal |
| 面向对象 | 将代码组织为具有状态和行为的相互作用对象 | Smalltalk、C++、Java、Python |
| 函数式 | 描述*计算什么*：纯函数、不可变数据、无副作用 | LISP、Haskell、Erlang、Clojure |
| 声明式 | 指定期望的结果，而不是执行过程 | SQL、Prolog、HTML |
| 逻辑式 | 定义事实与规则；让推理引擎找到答案 | Prolog、Datalog |
| 并发 / Actor | 将程序建模为相互通信的进程或 actor | Erlang、Go、Elixir |
| 响应式 | 将程序建模为对事件做出响应的数据流 | RxJS、Elm、Kotlin Arrow |
| 元编程 | 编写生成或变换其他代码的代码 | Lisp 宏、Rust 宏、Julia |

---

## 命令式编程

最古老的范式：程序是修改状态的指令序列，机器逐条执行语句并更新变量和内存。汇编语言是纯命令式的；C 和 Pascal 引入了结构化控制（循环、条件、函数），但并未放弃状态变化模型。

**优势：** 直接映射到硬件，最大程度的控制，性能可预测。
**劣势：** 难以推理全局状态，易出错的变量修改，难以并行化。

**代表语言：** Assembly、Fortran、COBOL、C、Pascal、BASIC

---

## 面向对象编程（OOP）

OOP 将代码组织为*对象*——状态（字段）和行为（方法）的集合体。对象通过发送消息（调用方法）来通信。这一范式诞生于 Simula（1967），在 Smalltalk（1972）中得到完善，并由 C++、Java、Python 广泛普及。

**核心概念：**
- **封装** — 隐藏内部状态，暴露受控接口
- **继承** — 从父类复用行为，在子类中进行专化
- **多态** — 相同接口在运行时作用于不同类型
- **消息传递** — Alan Kay 的原始愿景：对象通信而不是共享内存

<Callout type="warning">
  OOP 因助长偶然复杂性而受到批评：深层继承层次、隐藏在对象内部的共享可变状态，以及"香蕉-大猩猩-丛林"问题（你只想要香蕉，却得到了抱着香蕉的大猩猩和整片丛林）。现代语言通常更倾向于组合而非继承。
</Callout>

**代表语言：** Simula、Smalltalk、C++、Java、C#、Python、Ruby、Swift、Kotlin

---

## 函数式编程（FP）

函数式编程将计算视为数学函数的求值。关键约束：纯函数（无副作用，相同输入始终产生相同输出）、不可变数据（数据无法就地修改，只能变换为新数据）、一等函数（函数可作为参数传递、从函数返回，以及存储在数据结构中）。

**核心概念：**
- **纯函数** — 无全局状态、无 I/O、完全确定性
- **不可变性** — 数据结构不可修改；变换操作返回新值
- **高阶函数** — `map`、`filter`、`reduce` 作为构建块
- **递归** — 以递归调用表达迭代，而不是循环
- **引用透明性** — 任何表达式都可以被其值替换而不改变程序行为

**为何在今天仍然重要：**
- 并发更简单：不可变数据可以安全地在线程间共享
- 测试更简单：纯函数没有隐藏依赖
- 推理更简单：可以独立理解一个函数

**代表语言：** LISP、Haskell、ML、OCaml、Erlang、Clojure、Elm、F#、Scala（多范式）

## 声明式编程

声明式编程描述结果*是什么*，而不是*如何*计算。执行引擎负责确定步骤。SQL 是使用最广泛的声明式语言：`SELECT name FROM users WHERE age > 30` 描述了期望的结果；数据库查询规划器决定如何检索数据。

HTML 和 CSS 是声明式的：你描述结构和样式，浏览器决定如何渲染。Prolog 是声明式的：你陈述事实和规则，推理引擎推导结论。

**优势：** 简洁，接近问题领域，更易优化（运行时可选择最佳算法）。
**劣势：** 对性能控制较少；当引擎做了意外的事情时调试更困难。

**代表语言：** SQL、Prolog、HTML/CSS、Datalog、Terraform HCL、GraphQL

---

## 逻辑式编程

逻辑式编程是声明式编程的一种特殊形式，程序以逻辑陈述表达：事实（什么是真的）和规则（可以推断什么）。运行时使用合一（unification）和回溯（backtracking）来找到解答。

Prolog（1972）是典型的逻辑语言。你定义：
```prolog
parent(tom, bob).
parent(bob, ann).
ancestor(X, Y) :- parent(X, Y).
ancestor(X, Y) :- parent(X, Z), ancestor(Z, Y).
```
然后查询：`?- ancestor(tom, ann).` → `true.`

逻辑式编程在专家系统和自然语言处理领域找到了工业应用。Datalog 是 Prolog 的受限子集，为现代程序分析工具（如 Doop、Souffle）提供了动力。

**代表语言：** Prolog、Datalog、Mercury、Curry

## 并发与 Actor 模型编程

随着多核硬件的普及，如何表达并发程序变得至关重要。出现了两种主导模型：

**共享内存并发**（线程 + 锁）：多个线程访问同一内存；正确性依赖于谨慎的同步。这是 Java、C++ 和大多数 OS 级编程的模型。功能强大，但容易出错：竞态条件、死锁和优先级反转都是出了名的难以调试。

**消息传递 / Actor 模型**：进程（actor）之间不共享任何内容，仅通过发送消息通信。如果你想要另一个 actor 的数据，你必须请求它；你无法直接"伸手取走"。这从设计上消除了竞态条件。Erlang 在 1986 年开创了这一模型；Go 采用了带有 goroutine 和 channel 的相关模型；Elixir 以现代语法带来了这一模型。

| 模型 | 语言 | 优势 |
|------|------|------|
| 线程 + 锁 | C、Java、C++ | 细粒度控制 |
| Goroutine + Channel | Go | 简洁，结构化并发 |
| Actor | Erlang、Elixir、Akka | 故障隔离，分布式 |
| Async/Await | Python、JavaScript、Rust | 单线程协作式多任务 |
| CSP（通信顺序进程）| Go、occam | 形式化模型，可组合 |

## 响应式编程

响应式编程将计算建模为数据流和变化的传播。不是命令式地更新状态，而是定义关系："这个值依赖于那个值。"当源发生变化时，依赖值自动更新。

事件驱动的 UI 是天然的应用场景：当用户点击时，一系列点击事件通过变换（filter、map、debounce）流动并产生 UI 更新。RxJS、Reactor 和 RxJava 将响应式流引入 JavaScript 和 JVM。Elm 开创了 Elm 架构（Model-View-Update），这种纯函数式响应式模式影响了 React 和 SwiftUI。

**代表库/语言：** RxJS、RxJava、Reactor、Elm、Cycle.js、Kotlin Arrow

## 元编程

元编程是编写生成、检查或变换其他代码的代码。它弥合了语言与用该语言编写的程序之间的鸿沟。

**元编程的形式：**

- **宏**（Lisp、Rust、Julia）— 在编译时运行的代码，在编译前变换语法。Lisp 宏以代码即数据（同像性，homoiconicity）著称。Rust 过程宏驱动 `#[derive(Debug)]`。
- **反射**（Java、Python）— 在运行时检查类型、方法和字段。驱动依赖注入框架。
- **模板**（C++）— 编译时泛型编程，可以生成高度优化的专化代码；也因难以正确使用而臭名昭著。
- **代码生成** — 工具（如 `protoc`、`sqlc`）从模式（schema）生成源代码，减少样板代码。

**代表语言：** LISP、Scheme、Rust、Julia、C++（模板）、Ruby（eval）、Python（装饰器）

## 多范式语言

现代语言很少只坚持一种范式。它们将范式作为工具提供，让程序员为每个问题选择正确的模型：

| 语言 | 范式 |
|------|------|
| Python | 命令式、面向对象、部分函数式 |
| Scala | 面向对象 + 函数式（JVM） |
| Rust | 命令式、函数式特性、所有权系统 |
| Kotlin | 面向对象 + 函数式（空安全、lambda、协程） |
| JavaScript | 命令式、面向对象（基于原型）、函数式（一等函数） |
| Julia | 命令式、多重分派（一种多态形式）、元编程 |
| F# | 函数式优先，与 .NET 的 OOP 互操作 |

2010 年以来的趋势很明显：即使是传统的 OOP 语言（Java、C#、Kotlin、Swift）也采纳了函数式特性——lambda、不可变数据类、模式匹配、管道运算符。各范式正在融合。

## 如何选择范式

没有哪种范式适合所有问题。一个粗略的启发：

- **系统编程**（OS、驱动、嵌入式）→ 命令式（C、Rust、Zig）
- **业务逻辑、CRUD** → 面向对象或多范式（Java、C#、Python）
- **数据流水线、变换** → 函数式（Haskell、Clojure、F#）或混合（Scala、Kotlin）
- **并发分布式系统** → Actor 模型（Erlang、Elixir）或 CSP（Go）
- **数据查询** → 声明式（SQL、Datalog）
- **约束满足、AI 推理** → 逻辑式（Prolog、Datalog）
- **UI 事件处理** → 响应式（Elm、RxJS、SwiftUI）
